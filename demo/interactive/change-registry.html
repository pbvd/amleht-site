Post-mortem — what happened and what changed

On Friday 8 May 2026 at approximately 20:00 CEST, a Nebius Support Team advisory arrived in the Gmail inbox (`paul.bloemenveld@gmail.com`) disclosing CVE-2026-31431, a critical Linux local privilege escalation vulnerability affecting the `algif_aead` kernel module. The Nebius admin server at 89.169.114.174 was immediately identified as a potentially affected host.

Phase 0 pre-flight checks caught a host identity discrepancy — the task assumed an orderer VM existed as a separate instance. In reality only one Nebius VM existed at the time. The pipeline stopped, the discrepancy was documented honestly, and the classification was corrected before proceeding.

Phase 1 mitigation applied a modprobe.d block disabling `algif_aead` permanently. Verification confirmed the module was absent from `lsmod`. The orderer process was correctly noted as not present on the admin server.

Phase 2 classification recorded the incident as a DORA Art.28 TPR incident with `ledger_status: PENDING` — Agent XII attestation was deferred because the binary did not exist. This was recorded honestly rather than fabricated.

The post-mortem produced the automated incident pipeline (v1.0.0 then v1.1.0), 24/7 Gmail security watch on the Nebius VM via systemd, OAuth 2.0 authentication, and provisioning of the orderer VM. The compliance chain will be sealed retrospectively once Agent XII CLI is built.

Conclusion and governance narrative

This change record was triggered by a live CVE response and grew to encompass the full architectural gap identified during that response. The pre-flight discipline — refusing to write a governance record until technical facts were verified — prevented three material errors: committing false architecture documentation, merging a branch that would have deleted 143,000 lines, and deploying an HTML file as a binary.

What the incident revealed was not a failure but a natural consequence of building at pace. The orderer VM existed in architecture documentation before it existed in infrastructure. The Agent XII binary existed as a library before it existed as an executable. Both gaps are now formally documented with remediation paths.

The three deferred items have clear, honest scope: Zoho IMAP awaits a support response; Agent XII CLI is a full sprint; retrospective attestation follows automatically once the CLI exists. No item is deferred because the work was avoided — each is deferred because the dependency does not yet exist.

The five delivered items represent a material improvement in amleht's security and operational posture: automated 24/7 incident detection, sovereign EU alerting infrastructure, a provisioned orderer VM, post-quantum SSH on all hosts, and ML-DSA-65 dual-signing already active on the compliance chain signing path.
